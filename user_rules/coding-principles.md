# 通用编码原则

## 1. 先想后写

理解需求后，先给出高层方案（2-3 句），再动手改代码。

- 陈述你的假设，不确定时提问
- 有多个解读时呈现出来，不默默选一个
- 有更简单的方案要说出来
- 不清楚就停下来，说清楚哪里不清楚，然后问

## 2. 简单优先

用最少的代码解决问题，不写推测性代码。

- 不超需求添加功能
- 不为单次使用创建抽象
- 不为"可能的未来需求"写代码
- 不给不可能发生的场景加错误处理
- 如果 200 行能解决的事写了 50 行就做到了，就写 50 行

## 3. 小步快跑

每改完一个独立功能点就验证，不攒到最后。

- 第一版让功能工作，第二版才考虑性能和美感
- 改完一小步 → 验证 → 再改下一步
- 不把风险攒到最后一次性暴露

## 4. 不重复造轮子

先查项目里、标准库里、框架里是否已有现成工具和命令。

- 先查 `~/.trae/roles-memory/patterns/verified-solutions.md`
- 先查项目现有代码，不要重复实现已有功能
- 能用标准库的不用第三方库，能用第三方库的不自己写

## 5. 管道思维

复杂逻辑拆成多个单一职责的小步骤，可测试、可复用。

- 一个函数只做一件事
- 步骤之间通过明确的输入/输出连接
- 每个步骤可以独立验证

## 6. 上下文感知

读相关文件理解现有模式，新代码匹配现有风格。

- 打开目标文件，读上下文，再动手改
- 匹配现有代码风格，不是你的个人偏好
- 不改不改的不相干的代码

## 7. 边界处理

必须处理的边界情况：

- 异步操作：超时和错误处理
- 用户输入：验证和清理
- 空值/null：显式处理
- 文件路径：空格、特殊字符

## 8. 外科手术式改动

只碰必须碰的，只清理自己弄乱的。

- 不顺手"优化"相邻代码、注释、格式
- 不重构没坏的东西
- 匹配现有风格，即使你觉得你的方式更好
- 如果改动产生了孤立代码（无用的 import/变量/函数），清理掉
